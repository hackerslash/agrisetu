Bedrock Integration Plan for AgriSetu Voice AI
Date: 2026-03-02

Objective
- Replace the current generic AI extraction dependency with an Amazon Bedrock-centered architecture.
- Use a GPT 120B-class model as the primary extraction model for voice order understanding.
- Add Bedrock-native capabilities (Knowledge Bases for RAG, Data Automation, and Flows) to improve accuracy, explainability, and operational control.

Current State Analysis (from codebase)
1) Mobile voice capture and submission
- Mobile records audio in apps/mobile/lib/features/voice/screens/voice_order_screen.dart.
- Audio is sent to POST /api/v1/farmer/voice/parse-order using multipart form data through apps/mobile/lib/core/api/api_client.dart.
- Voice result expects structured extraction: cropName, quantity, unit, matchedGigId, confidence, clarification question.

2) Backend voice endpoint and orchestration
- Voice endpoint is in apps/api/src/routes/farmer.ts at /voice/parse-order.
- Current sequence:
  a) Validate request.
  b) If audio is provided, transcribe with AWS Transcribe via apps/api/src/services/transcribe.ts.
  c) Build gig context for that farmer (serviceable gigs, bounded list).
  d) Extract order intent via apps/api/src/services/ai-order-parser.ts.
  e) Return structured extraction + metadata to mobile.

3) Current AI extraction implementation
- ai-order-parser currently calls a generic chat-completions endpoint using BASE_URL, API_KEY, MODEL_ID environment variables.
- If model call fails, parser falls back to deterministic heuristics (regex + unit aliases + gig matching score).
- This means AI is already abstracted enough to migrate without changing mobile contracts.

4) Existing AWS footprint
- AWS Transcribe is already integrated and production-shaped (S3 upload, transcription job polling, cleanup).
- AWS S3 is already used for documents and avatar pipelines.
- This reduces integration risk for Bedrock expansion.

5) Main gaps today
- No Bedrock model governance or routing.
- No retrieval-augmented context beyond in-request gig list.
- No AI orchestration graph (flows) for confidence-based routing.
- No managed ingestion pipeline for continuously updating domain knowledge.

Target Architecture (Bedrock-first)
Primary model strategy
- Main model: GPT 120B-class model on Amazon Bedrock as the default extraction model.
- If exact GPT 120B artifact is not directly available in a given account/region, onboard equivalent 120B-class model through the Bedrock-supported onboarding path and keep the same inference interface.
- Keep deterministic fallback parser in place as a final guardrail.

High-level architecture
1) Mobile App
- Record audio and call the same /voice/parse-order API (no contract break).

2) API Layer (Node/Express)
- Keep /voice/parse-order endpoint.
- Keep current authentication and farmer context resolution.
- Replace direct generic model call with Bedrock inference adapter.

3) Speech-to-text stage
- Continue Amazon Transcribe as first stage (already integrated and working).
- Store transcript and detected language as current context for downstream extraction.

4) Bedrock Flow (orchestration layer)
- Step A: normalize transcript and language metadata.
- Step B: fetch retrieval context from Bedrock Knowledge Base.
- Step C: invoke GPT 120B-class model for structured extraction.
- Step D: run policy checks (unit normalization, quantity bounds, allowed gig IDs).
- Step E: branch:
  - high confidence and valid: return actionable extraction.
  - medium confidence: return clarification question.
  - low confidence or malformed output: invoke fallback extraction path.

5) Bedrock Knowledge Base (RAG)
- Index business/domain data relevant to extraction quality:
  - Published gig catalog snapshots (crop, variety, unit, min quantity, vendor region constraints).
  - Unit and synonym dictionaries (multilingual).
  - Historical transcript-to-confirmed-order pairs (privacy-safe, anonymized).
  - Crop taxonomy and common regional naming variations.
- Retrieval returns top contextual chunks that are injected into extraction prompts.

6) Bedrock Data Automation (ingestion and structuring)
- Use Data Automation pipelines to continuously transform new source data into KB-ready documents:
  - Vendor catalog updates.
  - New agricultural product metadata and synonym sets.
  - Curated domain documents.
- Push processed outputs to S3 and refresh Knowledge Base indexes on schedule.

7) Observability and governance
- Track model latency, confidence distribution, clarification rate, and fallback rate.
- Add trace IDs from mobile request to Bedrock invocation and final response.
- Add prompt/version metadata for reproducibility and rollback.

Reference request flow
- Mobile audio -> API /voice/parse-order
- API -> AWS Transcribe -> transcript
- API -> Bedrock Flow
  - Flow -> Bedrock Knowledge Base retrieval
  - Flow -> GPT 120B-class extraction
  - Flow -> validation + confidence gate
- API -> returns extraction payload in current schema
- Mobile -> prefilled order confirmation screen

How each Bedrock feature is leveraged
1) Bedrock model inference
- GPT 120B-class model is the primary decision engine for intent extraction and disambiguation.

2) Bedrock Knowledge Bases (RAG)
- Grounds model output in current gig inventory, units, regional terms, and catalog constraints.
- Reduces hallucinated products/units and improves matchedGigId accuracy.

3) Bedrock Data Automation
- Automates transformation of raw domain artifacts into structured knowledge docs.
- Reduces manual curation overhead and keeps KB fresh.

4) Bedrock Flows
- Encodes end-to-end orchestration with branching logic and confidence gates.
- Makes pipeline behavior explicit, testable, and maintainable.

Implementation Plan (no-code rollout)
Phase 0: Readiness and design (1 week)
- Freeze extraction response schema as contract.
- Define confidence thresholds and acceptance criteria.
- Define model routing policy and fallback policy.

Phase 1: Bedrock adapter layer (1 week)
- Introduce provider abstraction in backend (current generic call -> Bedrock adapter).
- Keep legacy model path behind feature flag for rollback.

Phase 2: Knowledge Base foundation (1-2 weeks)
- Build KB datasets from gigs + domain dictionaries.
- Configure retrieval settings (top-k, metadata filters by state/unit/crop).
- Add offline evaluation using historical transcripts.

Phase 3: Data Automation pipelines (1 week)
- Configure ingestion sources, transformation rules, and refresh cadence.
- Add quality checks (dedupe, stale data detection, schema checks).

Phase 4: Bedrock Flow orchestration (1 week)
- Implement confidence-gated orchestration.
- Add hard validation post-model (unit, quantity, matchedGig existence).

Phase 5: Progressive rollout (1-2 weeks)
- Canary by small traffic percentage.
- Compare against baseline on:
  - actionable extraction rate
  - clarification rate
  - fallback rate
  - average latency
  - manual correction rate on order confirmation screen
- Ramp to 100% after thresholds are met.

Phase 6: Optimization and scale
- Add caching for repeated retrieval contexts.
- Add language-specific prompt variants.
- Tune prompt and retrieval by crop/region cohorts.

Data and prompt strategy
- Keep output strictly structured to current mobile/backend schema.
- Enforce allowed units: kg, quintal, ton, bag, litre.
- Require matchedGigId to come only from retrieved/available gig IDs.
- Ask one concise clarification question when confidence is insufficient.

Security and compliance controls
- IAM least privilege for Bedrock, S3, and KB access.
- Encrypt all S3 buckets and Bedrock-related data stores.
- Redact PII from transcripts before long-term analytics storage.
- Set retention policy for raw audio and transcripts.

Success metrics (must-meet)
- +15% actionable extraction rate versus current baseline.
- -30% clarification prompts for common crops and units.
- -40% fallback parser usage.
- p95 voice-to-extraction latency under agreed product threshold.
- No increase in malformed extraction payloads.

Risks and mitigations
1) Model availability variance by region/account
- Mitigation: enforce model ID indirection and approved fallback model list.

2) Retrieval drift or stale KB
- Mitigation: scheduled refresh + freshness checks + canary validation.

3) Latency increase from RAG + orchestration
- Mitigation: compact retrieval contexts, tuned top-k, and flow step optimization.

4) Incorrect matched gig under ambiguous transcript
- Mitigation: confidence gate + explicit clarification + deterministic post-validation.

Final recommendation
- Keep the existing mobile contract unchanged.
- Migrate backend AI to Bedrock in layers: Model -> RAG -> Data Automation -> Flows.
- Use GPT 120B-class model as the primary model, with policy-driven fallback and strict validation.
- Treat rollout as an observable, reversible migration with canary gates.
